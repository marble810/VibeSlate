# Marble Panel — Docker Deployment

Docker is the unified deployment platform for Marble Panel.
Run `bun run docker:init`, choose LAN or Public mode, then `bun run docker:up`.

## Architecture

```text
bun run docker:init
  → Choose LAN or Public
  → Generate Docker runtime config in data/docker/
  → Same Docker image for both modes

LAN mode:
  Client device → Docker app container (:12001)
  Direct HTTPS with mkcert certificates
  QR code certificate installation via bun run lan:https

Public mode:
  Internet client → Caddy (:80/:443) → Docker app container (internal :12001)
                                      → Fail2Ban (auth rate limiting)
```

## Quick Start

```bash
# 1. Initialize — choose lan or public
bun run docker:init

# 2. Start the stack
bun run docker:up

# 3. Verify
bun run docker:smoke -- --mode lan    # or --mode public
```

## Commands

| Command | Description |
|---|---|
| `bun run docker:init` | Interactive setup — choose LAN or Public mode |
| `bun run docker:up` | Start containers based on init configuration |
| `bun run docker:smoke -- --mode <lan\|public>` | End-to-end verification |

## Modes

### LAN Mode

For trusted devices on the same local network. The app container exposes a host port,
and devices connect directly via HTTPS with mkcert-generated certificates.

```bash
bun run docker:init   # Choose "lan"
bun run docker:up     # Starts app container on port 12001
```

Certificate installation (run on the host machine, not in the container):

```bash
bun run lan:https     # QR code + TUI for cert installation on mobile devices
```

Features:
- Direct HTTPS access (no reverse proxy)
- mkcert + QR code certificate installation
- Password protection optional (prompted during init)
- No Caddy, no Fail2Ban, no hidden entry gate

### Public Mode

For public internet deployment with security hardening.

```bash
bun run docker:init   # Choose "public"
bun run docker:up     # Starts app + Caddy + Fail2Ban
```

Requirements:
- A domain name pointed to your server
- Password protection (mandatory)
- Optional: hidden entry gate (discovery-reduction layer)

Features:
- Caddy terminates TLS and strips spoofed headers
- App container has no host port mapping (internal only)
- Fail2Ban monitors auth failures and bans IPs
- Banned IPs are blocked by the app on every request
- Optional hidden entry path (root returns 404)

## Configuration

All configuration lives in `data/docker/` and is excluded from version control:

| File | Purpose |
|---|---|
| `data/docker/mode` | "lan" or "public" |
| `data/docker/.env` | Non-secret Compose mode variables |
| `data/docker/server.config.json` | App server config (mounted read-only) |
| `data/docker/certs/` | TLS certificates (LAN mode, mounted read-only) |
| `data/docker/caddy/Caddyfile` | Caddy configuration (Public mode) |

## Prerequisites

- Docker + Docker Compose v2
- **LAN mode**: mkcert (`brew install mkcert` / `apt install mkcert`)
- **Public mode**: A domain name with DNS pointing to your server

## Verification Checklist

After starting, verify with `bun run docker:smoke -- --mode <mode>` or manually:

| Check | LAN | Public |
|---|---|---|
| App container running | ✓ | ✓ |
| App responds on internal :12001 | ✓ | ✓ |
| Host port mapping | ✓ | ✗ (by design) |
| Caddy running | ✗ | ✓ |
| Fail2Ban running | ✗ | ✓ |
| Caddy → app connectivity | ✗ | ✓ |
| Config mounted | ✓ | ✓ |
| TLS certs mounted | ✓ | ✗ |

## Directory Layout

```text
marble-panel/
├── .dockerignore               # Root-level (Docker build context requires it here)
├── docker/
│   ├── Dockerfile              # Builds frontend + bundles server
│   ├── docker-compose.yml      # Unified compose (all services)
│   ├── docker-compose.lan.yml  # LAN override (host port mapping)
│   ├── caddy/Caddyfile         # Reference Caddyfile
│   └── fail2ban/               # Fail2Ban config (Dockerfile + jail/filter/action)
├── scripts/
│   ├── docker-init.ts          # Interactive init
│   ├── docker-up.ts            # Mode-aware launcher
│   ├── docker-smoke.ts         # Smoke tests
│   └── lib/lan-tls.ts          # Shared TLS certificate utilities
└── data/docker/                # Generated by docker:init (gitignored)
    ├── mode
    ├── .env
    ├── server.config.json
    ├── certs/                  # LAN mode
    └── caddy/Caddyfile         # Public mode
```

## Security

- **No secrets in images or compose env**: Provider tokens, password hashes, and
  session tokens are loaded from the mounted server config, never baked into the
  Docker image or passed through Compose environment variables.
- **LAN mode**: root CA private key (`rootCA-key.pem`) is never mounted into the
  container. Only the leaf cert and root CA public cert are exposed.
- **Public mode**: App container has no host port mapping — only Caddy can reach it.
- **Fail2Ban**: Ban list is written to a shared volume and checked by the app on
  every request. Banned IPs receive 403 Forbidden.
- **Trusted proxies**: X-Forwarded-For headers are only accepted from Docker
  bridge CIDRs. Spoofed headers are rejected.

## Troubleshooting

### docker:init fails with "mkcert not found"

Install mkcert:
- macOS: `brew install mkcert`
- Linux: `apt install mkcert` or `pacman -S mkcert`
Then run `mkcert -install` to create the local CA.

### docker:up starts but app is not reachable

```bash
# Check app container logs
docker compose logs app

# For Public mode, check Caddy
docker compose logs caddy

# Verify the mode file
cat data/docker/mode
```

### Switching modes

Re-run `bun run docker:init` and choose the other mode. Existing containers
must be stopped first:

```bash
docker compose down
bun run docker:init   # Choose the new mode
bun run docker:up
```
