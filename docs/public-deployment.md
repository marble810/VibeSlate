# Marble Panel — Public Mode Security Add-ons

Public mode extends the Docker deployment with security hardening for public
internet exposure. These add-ons are only active when you choose "public" during
`bun run docker:init`.

> For the full Docker deployment guide (both LAN and Public), see
> [docker-deployment.md](./docker-deployment.md).

## Architecture (Public Mode Add-ons)

```text
Internet Client → Caddy (TLS termination) → App (internal only)
                       ↓                           ↓
                 Strips spoofed              Checks Fail2Ban
                 IP headers                  ban list
                                                  ↑
                                         Fail2Ban (reads auth log,
                                           writes ban list)
```

## What Public Mode Adds

| Component | Purpose |
|---|---|
| **Caddy** | Public TLS termination, strips spoofed headers, sets trusted `X-Forwarded-For`, SSE-compatible reverse proxy |
| **Fail2Ban** | Reads auth failure logs from shared volume, writes banned IPs to shared file |
| **App ban check** | Reads Fail2Ban ban list on every request, blocks banned IPs |
| **Trusted proxy enforcement** | Only accepts forwarded headers from Docker bridge CIDRs |
| **Mandatory password auth** | App refuses to start in Public mode without a password hash |
| **Hidden entry gate** (optional) | Root path returns 404, app only accessible via configured path |

## How to Enable

```bash
bun run docker:init   # Choose "public" when prompted
bun run docker:up     # Starts app + Caddy + Fail2Ban
```

## Caddy

Caddy is the sole public entry point. It:
- Terminates TLS with Let's Encrypt (automatic via Caddy)
- Strips any `X-Forwarded-For` headers from untrusted clients
- Sets `X-Forwarded-For` and `X-Real-IP` to the real client IP
- Disables response buffering for SSE streams (`flush_interval -1`)
- Logs access in JSON format to `/var/log/caddy/marble-panel-access.log`

The Caddyfile is generated by `docker:init` with your domain name and stored
at `data/docker/caddy/Caddyfile`.

## Fail2Ban

Fail2Ban monitors auth failure logs from the app and bans IPs that exceed
the threshold:

- **5 failures** in **10 minutes** → **10-minute ban**
- Ban list is written to a shared volume file (`/var/lib/fail2ban/marble-panel-banned.txt`)
- The app reads this file on every request and returns 403 for banned IPs

### Verification

```bash
# Check jail status
docker compose exec fail2ban fail2ban-client status marble-panel

# Trigger a ban (send 5 wrong passwords quickly)
for i in $(seq 1 5); do
  curl -X POST https://YOUR_DOMAIN/auth/login -d 'password=wrong'
done

# Check banned IPs
docker compose exec fail2ban fail2ban-client status marble-panel | grep "Banned IP list"

# Verify the ban blocks traffic
curl -v https://YOUR_DOMAIN/   # Should return 403
```

## Trusted Proxy Verification

In Public mode, the app only trusts forwarded headers from Docker bridge IPs:

```bash
# 1. Spoofed header from untrusted source must be ignored
curl -X POST https://YOUR_DOMAIN/auth/login \
  -H 'X-Forwarded-For: 1.2.3.4' \
  -d 'password=wrong'
docker compose logs app | grep MARBLE_AUTH_FAIL
# → ip= should be Caddy's IP, NOT 1.2.3.4

# 2. Real client IP appears in logs
curl -X POST https://YOUR_DOMAIN/auth/login -d 'password=wrong'
# → ip= should be your actual public IP
```

## Hidden Entry Gate (Optional)

Adds a discovery-reduction layer. The app becomes accessible only through
a configurable path:

- `https://YOUR_DOMAIN/` → 404 Not Found
- `https://YOUR_DOMAIN/anything` → 404 (no app shell leaks)
- `https://YOUR_DOMAIN/<hidden-path>/` → App
- API endpoints (`/auth/*`, `/events`, `/api/*`) and static files are always
  accessible regardless of path
- Login redirects go to `/<hidden-path>/` instead of `/`
- PWA `start_url` and `scope` are set to `/<hidden-path>/` (requires rebuild
  with `MARBLE_HIDDEN_ENTRY_PATH`)

Enable during `docker:init` when prompted, or set manually in
`data/docker/server.config.json`:

```json
{
  "hidden_entry": {
    "enabled": true,
    "path": "marble-7a2f",
    "root_response": "404"
  }
}
```

Rebuild the container after changing the hidden path to ensure PWA manifests match.

## Log Paths

| Component | Location |
|---|---|
| App stdout/stderr | `docker compose logs app` |
| Auth failures | `/var/log/marble-panel/auth.log` (shared volume) |
| Caddy access | `/var/log/caddy/marble-panel-access.log` (inside container) |
| Fail2Ban log | `/var/log/fail2ban.log` (inside container) |
| Ban list | `/var/lib/fail2ban/marble-panel-banned.txt` (shared volume) |

## Security Notes

- **App isolation**: In Public mode, the app container has no host port mapping.
  Only Caddy can reach it over the internal Docker network.
- **Trusted proxies**: `X-Forwarded-For` and `X-Real-IP` are only trusted from
  IPs in the configured CIDR list. Spoofed headers from internet clients are
  stripped by Caddy and rejected by the app.
- **Startup guard**: The app refuses to start in Public mode if password auth is
  disabled, `password_hash` is missing, or no trusted proxy CIDR is configured.
- **No secrets in logs**: Auth failure logs contain only IP, path, and reason.
  Passwords, tokens, cookies, and certificate keys never appear.

## Troubleshooting

### App refuses to start in Public mode

The app requires Public mode to be fail-closed:
- `auth.enabled` must be `true`
- `auth.password_hash` must be set
- `public.trusted_proxies` must contain at least one trusted proxy IP/CIDR

Re-run `bun run docker:init` to regenerate a valid Public config.

### Caddy "connection refused" to app

```bash
docker compose exec caddy wget -qO- http://app:12001/
```

### Fail2Ban not banning

```bash
# Check jail has the correct log file
docker compose exec fail2ban fail2ban-client status marble-panel | grep "File list"

# Verify auth failures are being logged
docker compose exec app cat /var/log/marble-panel/auth.log
```

### SSE streaming issues

Caddy uses `flush_interval -1` to disable response buffering. If SSE still lags:
1. Check Caddy version ≥ 2.7
2. Verify the Caddyfile has `flush_interval -1` inside the `reverse_proxy` block
3. The app's `idleTimeout` is set to `0` (keeps SSE connections alive indefinitely)
